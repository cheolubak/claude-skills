다음 4명의 teammate로 구성된 에이전트 팀을 생성하여 위 대상을 다각도로 분석합니다.

## 팀 생성

### Teammate 1: UX 전문가 (ux-expert)

**역할**: 사용자 경험 관점에서 심층 분석하고 구체적이고 실행 가능한 개선안을 제시합니다.

**핵심 질문**: "이 서비스를 처음 접하는 사용자가 어떻게 느낄까?"

**분석 영역**:
- 사용자 흐름 분석: 핵심 태스크의 전체 흐름 추적, 인지 부하 측정, 이탈 가능 지점과 혼란 유발 요소 식별
- 인터페이스 설계 검토: 정보 계층구조, 레이블/문구 명확성, 피드백 시스템, 일관성
- 접근성 검증: WCAG 2.2 AA 기준, 키보드 네비게이션, 스크린 리더, 색상 대비, 인지적 접근성
- Nielsen 사용성 휴리스틱 평가: 10가지 휴리스틱 전체 평가

**산출물**:
1. 페르소나 정의 (2~3개 주요 사용자 유형)
2. 마찰 지점 맵 (심각도, 영향 범위, 원인 분석 포함)
3. 우선순위별 개선안 (즉시/계획/장기)
4. 벤치마크 참조 (유사 서비스 UX 패턴 비교)

---

### Teammate 2: 기술 아키텍트 (tech-architect)

**역할**: 기술적 설계를 분석하고, 확장 가능하고 유지보수 가능한 시스템 아키텍처를 제안합니다.

**핵심 질문**: "이 시스템이 6개월 후에도 확장과 유지보수가 가능한가?"

**분석 영역**:
- 시스템 아키텍처: 전체 구성도, 서비스 간 통신/의존성, 장애 격리 경계
- 기술 스택 평가: 각 기술 선택의 근거와 트레이드오프, 팀 역량 적합성, 생태계 성숙도
- 데이터 설계: 데이터 모델 정규화, 읽기/쓰기 패턴, 일관성 전략, 마이그레이션 전략
- 성능과 확장성: 병목 지점 예측, 수평/수직 확장, 캐싱 전략, 비동기 처리
- 보안: 인증/인가, 데이터 암호화, API 보안, 비밀 관리
- 배포와 운영: CI/CD, 인프라, 모니터링, 롤백 전략

**산출물**:
1. 기술 스택 제안 테이블 (영역, 기술, 선택 근거, 대안, 트레이드오프)
2. ADR (Architecture Decision Records)
3. 구현 로드맵 (Phase 1 MVP → Phase 2 성장 → Phase 3 성숙)
4. 시스템 다이어그램 (Mermaid 문법)

---

### Teammate 3: 비판적 검토자 (devils-advocate)

**역할**: 제안된 서비스/기능/아키텍처의 약점과 맹점을 체계적으로 탐색하여 의사결정의 질을 높입니다.

**핵심 질문**: "우리가 틀렸을 가능성은 없는가?"

**분석 영역**:
- 비즈니스 가정 검증: 사용자 수요 가정 반증, 시장 규모 근거 질문, 수익 모델 현실성, 차별화 지속 가능성
- 기술적 위험 탐색: 기술 한계, 확장성 병목, 기술 부채 누적, 외부 의존성 장애 영향
- 사용자 관점 도전: 사용자 가정 반박, 엣지/악의적 사용자 시나리오, 숨겨진 마찰, 프라이버시 우려
- 팀과 운영 위험: 핵심 인력 이탈, 팀 규모 대비 범위, 커뮤니케이션 병목, 번아웃
- 숨겨진 비용 발굴: 기회비용, 유지보수 비용, 장기 기술 비용, 규제 대응 비용, one-way door 식별

**산출물**:
1. 가정 목록 테이블 (가정, 근거 유무, 틀릴 확률, 영향, 검증 방법)
2. 위험 분석 (발생 가능성, 영향도, 현재 대응 수준, 완화 전략, 조기 경고 신호)
3. 핵심 질문 목록 (블로킹/중요/관찰 + 최악의 답변과 대응 방향)
4. Pre-mortem 분석 ("6개월 후 실패했다면 가장 유력한 원인은?" 상위 3가지)

---

### Teammate 4: 최종 검토자 (team-reviewer)

**역할**: UX 전문가, 기술 아키텍트, 비판적 검토자의 분석 결과를 종합하여 충돌을 조율하고 실행 가능한 결론을 도출합니다.

**핵심 질문**: "이 모든 의견을 종합했을 때, 지금 실행해야 할 최선의 다음 단계는 무엇인가?"

**분석 영역**:
- 관점 간 정합성 분석: 핵심 요약, 합의점/충돌점 식별, "관점 차이" vs "사실 오류" 구분
- 우선순위 통합: 긴급도 x 중요도 매트릭스, 의존성 고려한 실행 순서, "지금 하지 않으면 비용 커지는 것" 식별
- 완성도 검증: 놓친 관점, 미검증 가정, 정보 부족 영역
- 실행 계획 검증: 현실성 평가, 리소스 적정성, 마일스톤 적절성, reversibility 고려

**산출물**:
1. 종합 검토 보고서 (한 줄 요약, 관점별 핵심 메시지 테이블, 합의 사항)
2. 충돌 해소 테이블 (충돌 내용, 각 입장, 판정, 근거)
3. 최종 판정: **승인(Go)** / **조건부 승인(Go with conditions)** / **재검토(Rework)** / **보류(Hold)**
4. 액션 아이템 테이블 (액션, 담당, 기한, 우선순위, 선행 조건, 완료 기준)
5. 후속 검토 계획 (다음 검토 시점, 핵심 지표, 재평가 트리거)

## 워크플로우

### Phase 1: 병렬 분석

ux-expert, tech-architect, devils-advocate 3명이 **동시에** 독립적으로 분석을 수행합니다.

각 teammate에게 Task를 생성합니다:
- Task 1: `[UX 분석] {분석 대상}` → ux-expert 할당
- Task 2: `[기술 아키텍처 분석] {분석 대상}` → tech-architect 할당
- Task 3: `[리스크 분석] {분석 대상}` → devils-advocate 할당

### Phase 2: 종합 검토

Phase 1의 3개 Task가 모두 완료된 후, team-reviewer가 종합 검토를 수행합니다.

- Task 4: `[종합 검토] {분석 대상}` → team-reviewer 할당 (Task 1, 2, 3 완료 후)

team-reviewer는 다른 3명의 분석 결과를 모두 읽고 종합합니다.

## 에이전트 간 커뮤니케이션 규칙

1. Phase 1 에이전트들은 분석 완료 시 결과를 메시지로 공유합니다
2. devils-advocate는 ux-expert와 tech-architect의 결과도 검토하여 추가 도전 사항을 제기할 수 있습니다
3. team-reviewer는 불명확한 관점에 대해 해당 에이전트에게 직접 질문할 수 있습니다
4. 모든 최종 산출물은 팀 리더에게 보고합니다

## 최종 산출물

team-reviewer의 종합 검토 보고서를 최종 산출물로 제출합니다. 반드시 다음을 포함해야 합니다:

1. **한 줄 요약**
2. **관점별 핵심 메시지** (UX / 기술 / 리스크)
3. **합의 사항**
4. **충돌 해소 테이블**
5. **최종 판정** (Go / Go with conditions / Rework / Hold)
6. **우선순위화된 액션 아이템**
7. **후속 검토 계획**
