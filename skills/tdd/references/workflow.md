# TDD 워크플로 상세

레드-그린-리팩터 사이클을 실제로 굴리는 방법. 스택 무관한 규율을 다룬다. 스택별 실전 예시는 [frontend-vitest.md](frontend-vitest.md), [backend-jest.md](backend-jest.md)를 본다.

## 테스트 리스트: 구현 전에 케이스를 나열한다

코드를 짜기 전에, 만들 동작을 **작은 케이스 목록**으로 적는다. 그리고 하나씩 정복한다. 새 케이스가 떠오르면 리스트에 추가만 하고 지금 하던 것을 끝낸다.

예) `formatPrice(amount, currency)` 를 만든다면:

```
[ ] 1000 → "₩1,000"
[ ] 0 → "₩0"
[ ] 음수 -1000 → "-₩1,000"
[ ] currency="USD" → "$1,000"
[ ] 소수점 있는 값 반올림
```

- 위에서부터 하나 골라 RED → GREEN → REFACTOR.
- 정복한 항목은 `[x]`로 체크. 한 번에 **하나만** 진행 중이어야 한다.
- 리스트가 비면 그 기능은 끝난 것이다.

## 삼각측량 (triangulation): 하드코딩 → 일반화

GREEN 단계에서 통과시킬 가장 쉬운 방법이 **하드코딩**이면 일단 하드코딩한다. 그다음 두 번째 케이스를 추가해 하드코딩을 깨뜨리고, 그제서야 일반화한다.

```typescript
// RED 1: add(2, 3) === 5
// GREEN 1: 하드코딩으로 통과
function add(a: number, b: number) {
  return 5;
}

// RED 2: add(10, 1) === 11  ← 하드코딩이 깨진다
// GREEN 2: 두 번째 케이스가 강제한 일반화
function add(a: number, b: number) {
  return a + b;
}
```

두 번째 점(케이스)이 생겨야 "선"(일반 로직)을 그을 근거가 생긴다. 케이스가 하나뿐일 때 성급히 일반화하지 않는다.

## 테스트 하나에 동작 하나 (one behavior per test)

- 한 `it`은 **한 가지 동작**만 검증한다. assert가 여러 개여도 같은 동작을 서술하면 괜찮지만, 서로 다른 동작을 한 테스트에 몰지 않는다.
- 테스트가 실패하면 **어떤 동작이 깨졌는지 이름만 보고** 알 수 있어야 한다.

## Arrange-Act-Assert (Given-When-Then)

각 테스트를 3구획으로 나눠 쓴다. 준비 → 실행 → 검증.

```typescript
it('재고가 0이면 품절로 표시한다', () => {
  // Arrange (Given): 준비
  const product = createProduct({ stock: 0 });

  // Act (When): 검증 대상 동작 실행
  const result = getStockLabel(product);

  // Assert (Then): 결과 검증
  expect(result).toBe('품절');
});
```

## 테스트 이름은 동작을 서술한다

- 이름은 **무엇을 하는지(동작·기대)** 를 서술한다. 구현이 아니라 관찰 가능한 결과를 쓴다.
- 나쁨: `it('works')`, `it('test create')`, `it('calls save')`
- 좋음: `it('빈 이름이면 검증 에러를 던진다')`, `it('존재하지 않는 id면 NotFoundException을 던진다')`

## tidy-first: 구조 변경과 동작 변경을 분리한다

한 커밋에 **구조 변경(리팩터)** 과 **동작 변경(기능/버그)** 을 섞지 않는다.

- **구조 변경**: 동작을 바꾸지 않고 코드 모양만 바꿈(이름 변경, 함수 추출, 이동). 테스트는 그대로 초록.
- **동작 변경**: 관찰 가능한 결과가 바뀜(새 기능, 버그 수정). 새/변경된 테스트가 동반.
- 리팩터는 **테스트가 초록일 때만** 한다. 빨간 상태에서 구조를 건드리면 무엇이 깨졌는지 알 수 없다.
- 순서 팁: 다음 기능을 넣기 어렵게 만드는 구조가 있으면, 먼저 구조를 정리(초록 유지)해 커밋하고, 그다음 기능을 추가한다.

## 커밋 리듬: 그린마다 작은 커밋

- 하나의 동작이 GREEN이 되고 리팩터까지 끝나면 **바로 커밋**한다. 커밋 단위가 작을수록 되돌리기 쉽다.
- 구조 변경 커밋과 동작 변경 커밋은 메시지로 구분한다. 예: `refactor: 가격 포맷 로직 추출` vs `feat: USD 통화 포맷 지원`.

```bash
# 한 사이클 예시
pnpm vitest run src/lib/format-price.test.ts   # GREEN 확인
git add -p
git commit -m "feat: 음수 금액 포맷 지원"
```

## 안티패턴

| 안티패턴 | 문제 | 대신 |
|---------|------|------|
| 구현 세부를 테스트 | 내부 함수 호출 여부·private 상태를 검증 → 리팩터할 때마다 테스트가 깨짐 | **관찰 가능한 동작**(반환값, DOM, 예외)을 검증 |
| 과도한 모킹 | 모든 협력자를 mock → 실제로는 아무것도 검증 못 하는 테스트 | 순수 로직은 mock 없이, 외부 경계(DB·네트워크)만 mock |
| 테스트 후행 작성 | 구현을 다 만든 뒤 테스트를 맞춤 → 이미 통과할 테스트만 나옴, 설계 피드백 없음 | **먼저 RED를 보고** 나서 구현 |
| 한 번에 여러 RED | 여러 테스트가 동시에 실패 → 어디부터 고칠지 혼란 | 테스트 리스트로 쪼개 **하나씩** |
| 리팩터를 빨간 상태에서 | 깨진 원인과 리팩터 실수가 뒤엉킴 | 초록으로 만든 **뒤에** 정리 |
